A field guide for working engineers

Bugs live at the edges.
Boundary value analysis finds them there.

Most defects don't hide in the middle of an input range — they hide at its borders, where a < should have been a <= and an off-by-one waits for production traffic. This guide covers the technique itself, why cyclomatic complexity tells you how much boundary testing a method actually needs, and how to put both to work in Java, mobile, and Angular codebases.

§ Foundations

What boundary value analysis is — and why it works

Boundary value analysis (BVA) is a black-box test design technique that selects test inputs at the edges of an input domain rather than sampling from its interior. If a field accepts ages 18–65, BVA says the interesting values are not 40 or 52 — they are 17, 18, 19, and 64, 65, 66. The technique rests on a well-worn empirical observation: implementations fail at partition edges far more often than inside them, because that's where developers write comparison logic — and comparison logic is where > vs. >= mistakes, off-by-one errors, and mishandled limits live.

The partner technique: equivalence partitioning

BVA is almost always taught and applied alongside equivalence partitioning (EP). EP divides the input space into classes the system should treat identically — for an 18–65 age gate, that's three partitions: below range (invalid), in range (valid), and above range (invalid). Any single value from a partition stands in for the whole class. EP tells you what to test; BVA tells you exactly which values to test — the ones sitting on each partition's border, where a misplaced boundary in the code would flip the behavior of the entire class.

Two-value vs. three-value BVA

The ISTQB syllabus distinguishes two flavors. Two-value BVA tests each boundary and its nearest neighbor across the border (e.g., 17 and 18 for a minimum of 18). Three-value BVA adds the neighbor inside the border as well (17, 18, 19), buying extra confidence for higher-risk domains — financial, medical, safety-critical — where you want proof the behavior is correct at the boundary and in its immediate vicinity on both sides.

Where boundaries hide beyond numeric ranges

Numbers are the textbook case, but any ordered domain has boundaries worth probing:

  • String lengths — a username of 3–20 characters means testing 2, 3, 20, and 21 characters, plus empty and null.
  • Collections — empty list, single element, exactly-at-capacity, one-over-capacity; first and last index of an array.
  • Dates and time — midnight, end of month, Feb 29, daylight-saving transitions, epoch rollovers, session-timeout instants.
  • Data types — Integer.MAX_VALUE, overflow points, precision limits of floating point, maximum payload sizes.
  • State and quota — zero balance, exactly-sufficient funds, last remaining seat, rate-limit thresholds.
The classic failure this catches: a requirement says "orders of $100 or more ship free." The developer writes total > 100. Every mid-range test passes — $50 pays shipping, $150 ships free. Only the boundary test at exactly $100.00 exposes that the single most talked-about value in the requirement is handled wrong.

Why teams adopt it: the economics

Exhaustive input testing is impossible; random sampling is inefficient. BVA concentrates a small, systematic set of test cases precisely where defect density is empirically highest. For a single bounded input, three-value BVA needs just seven values (min−1, min, min+1, a nominal interior value, max−1, max, max+1) to cover every transition a comparison operator can get wrong. That's the whole pitch: maximum defect-detection probability per test case written.

§ The Metric

The cyclomatic complexity connection

Cyclomatic complexity — McCabe's V(G) — is a white-box metric: the number of linearly independent paths through a piece of code's control-flow graph. Computed as V(G) = E − N + 2 (edges minus nodes plus two) or, more practically, as decision points + 1, it sets the minimum number of test cases required to exercise every independent path at least once — the foundation of basis path testing.

At first glance the two techniques live on opposite sides of the wall: BVA is black-box (designed from the specification), cyclomatic complexity is white-box (measured from the code). The connection is that they count the same things from different directions.

Every boundary is a decision; every decision is a boundary

Consider a range check: if (age >= 18 && age <= 65). To the complexity tool, the two comparisons are two decision points — with short-circuit evaluation counted, this single line contributes +2 to V(G). To the BVA practitioner, those same two comparisons are two boundaries, each demanding on-the-line and across-the-line test values. The predicates that inflate a method's cyclomatic complexity are, in large part, the very boundary conditions BVA targets. That gives you three practical bridges between the metric and the technique:

  1. V(G) sizes your test suite. Complexity gives the floor for path coverage; BVA supplies the specific input values that push execution across each predicate in both directions. A method with V(G) = 6 and mostly range-check predicates will need roughly its complexity's worth of boundary-driven cases — and the BVA table practically writes the parameterized test for you.
  2. V(G) prioritizes where BVA effort goes. High-complexity modules are statistically the defect-prone ones. When time is limited, run your complexity report (SonarQube, PMD, Checkstyle) and aim rigorous three-value BVA at the methods scoring highest — that's risk-based testing with a number behind it.
  3. BVA exposes what coverage numbers hide. You can hit 100% branch coverage on total > 100 with $50 and $150 — and never learn that $100 itself misbehaves. Path coverage says which roads were driven; boundary values say whether the road forks in the right place. You need both.
Rule of thumb for code review: when a method's cyclomatic complexity crosses ~10, it's not just "hard to read" — it has accumulated so many predicates (so many boundaries) that its BVA test matrix has become combinatorially expensive. Refactoring for lower V(G) is simultaneously refactoring for testable boundaries. Complexity thresholds and boundary test effort rise and fall together.
§ Java's Number Line

Type boundaries: where the numbers themselves end

Everything so far treated boundaries as something the specification creates — an age gate, a length limit. But in Java, every primitive numeric type imposes a second, silent set of boundaries that exist before any business rule is written. An int is not "a number"; it is exactly 232 numbers arranged in a fixed range, and arithmetic that steps outside that range doesn't fail — it wraps around silently. That makes even the simplest operation, a + b, a boundary-testing problem in its own right.

Addition alone has at least five boundary values

Take the plainest expression in the language: int c = a + b;. Before any requirement exists, the type system has already defined the values where this expression's behavior changes character. A credible boundary set for each operand:

#ValueWhy it's a boundary for a + b
1Integer.MIN_VALUEThe underflow seam. Adding any negative number wraps to a large positive result. Also the value with no representable negation.
2MIN_VALUE + 1The last value that survives negation; the smallest operand for which a + (−1) is still safe by one.
3−1 / 0 / +1The sign crossing. −1 + 1 flips all 32 bits to reach 0 (full carry propagation); 0 is the additive identity; ±1 are the smallest steps that can push either seam.
4MAX_VALUE − 1The last operand for which + 1 stays in range — the "min−1" of the overflow condition itself.
5Integer.MAX_VALUEThe overflow seam. MAX_VALUE + 1 is not an error — it is MIN_VALUE, silently.

This is BVA applied to the type rather than the spec — and it composes with the earlier sections: each of these values belongs in the same @ValueSource / @CsvSource tables, and overflow guards (if (a > MAX − b)…) are decision points that raise V(G) exactly like any business boundary.

What the wheel means for your code

// Silent by default: no exception, no warning — just the seam
int a = Integer.MAX_VALUE;
System.out.println(a + 1);              // -2147483648
System.out.println(Math.abs(Integer.MIN_VALUE)); // -2147483648 (still negative!)

// Promotion hides the smaller seams: byte + byte is int arithmetic
byte b1 = 100, b2 = 100;
int ok = b1 + b2;                        // 200 — fine as int…
byte wrapped = (byte) (b1 + b2);         // -56 — the seam returns at the cast

// Opt in to failure at the boundary — then test for it
@ParameterizedTest
@CsvSource({
    "2147483646, 1, false",  // MAX-1 + 1 → fits
    "2147483647, 1, true",   // MAX   + 1 → must throw
    "-2147483648, -1, true"  // MIN   - 1 → must throw
})
void addExactRespectsTheSeam(int a, int b, boolean overflows) {
    if (overflows) {
        assertThrows(ArithmeticException.class, () -> Math.addExact(a, b));
    } else {
        assertEquals((long) a + b, Math.addExact(a, b));
    }
}
The practical rules: money and anything user-accumulating gets long, Math.addExact/multiplyExact, or BigInteger/BigDecimal — and every overflow guard you write is a new predicate whose two sides your boundary table must exercise. If PIT's boundary mutator can turn your > into >= unnoticed, the seam is unguarded.

Boxed types: null joins the boundary set

Swap int for Integer and the value set grows by one — and it's the most dangerous member. A boxed type is a reference, so every boundary table gains a null row; and because Java auto-unboxes inside arithmetic, a + b stops being a pure computation and becomes an expression that can throw. The NullPointerException doesn't come from the + — it comes from the hidden a.intValue() the compiler inserts first. Which operands get unboxed depends on the mix:

ExpressionWhat the compiler emitsNull outcome
int + intPure primitive additionnull impossible — only the seam to test
Integer + intUnboxes the Integer, adds as intNPE if the boxed operand is null
Integer + IntegerUnboxes both, adds as int, re-boxes if assigned to IntegerNPE if either is null — two failure paths
flag ? 1 : boxedMixed branches promote the whole ternary to intNPE when the boxed branch is chosen while null — the classic ternary unboxing trap
 
// The null boundary in action
Integer a = null;
int b = 1;
int c = a + b;                // NPE — thrown by the hidden a.intValue(), not by "+"

// Boxing has its own boundary — and it sits exactly at the byte seam
Integer x = 127, y = 127;
Integer p = 128, q = 128;
System.out.println(x == y);   // true  — Integer cache covers [-128, 127]
System.out.println(p == q);   // false — one past the cache: identity != equality

// JUnit 5 ships the null boundary as a first-class source:
// a boxed parameter lets @NullSource extend the same value table
@ParameterizedTest
@NullSource
@ValueSource(ints = {Integer.MIN_VALUE, -1, 0, 1, Integer.MAX_VALUE})
void handlesFullBoundarySet(Integer amount) {
    if (amount == null) {
        assertThrows(NullPointerException.class, () -> service.credit(amount));
    } else {
        assertDoesNotThrow(() -> service.credit(amount));
    }
}
A boundary inside the boxing itself: the Integer.valueOf cache spans −128 to 127 — the byte range. Code that compares boxed values with == works perfectly in every test using small numbers and fails at 128. That's a three-value BVA target (127, 128, and −128/−129) hiding in what looks like infrastructure.

Null's real cost: paths, then combinatorics

Each nullable operand contributes twice to test burden. First, paths: the control-flow graph of Integer + Integer has an implicit exceptional edge per unboxing, and the moment you guard defensively — if (a == null || b == null) — you've added two decision points and raised V(G) by two. The white-box metric registers the reference boundary exactly as it registers a range check. Second, combinatorics: BVA per single operand is linear (seven values, or eight with null), but operands interact in arithmetic, so the honest test matrix is the cross product — and every nullable operand multiplies it by 8/7 while adding an entire failing row and column:

Test Matrix — a + b over int boundary values
TOGGLE BOXING AND WATCH NULL MULTIPLY THE TABLE
In range Overflow (wraps) NullPointerException

The matrix makes the trade explicit. Two primitive operands: 49 combinations, all of them value behavior. Two boxed operands: 64 — and 15 of them (23%) never reach your arithmetic at all, failing at unboxing instead. At four boxed operands the full product is 4,096, of which over 40% are NPE paths. Nobody runs the full product, which is exactly the point — the explosion is the argument for the standard mitigations:

  • Kill null at the perimeter. Objects.requireNonNull at the API boundary (or Bean Validation @NotNull) converts every interior null path into one tested fail-fast edge. Establishing the non-null invariant early collapses the 8-value interior matrix back to 7 — the cheapest combinatorial win available.
  • Keep arithmetic primitive. Unbox once at the entry point; do math in int/long. Boxed types are for optionality and collections, not for computation.
  • Test interactions pairwise. A pairwise covering array needs on the order of v² cases (~64 here) to cover every two-operand value combination regardless of how many operands exist — sane coverage of the interaction space without the full cross product.
  • Make tools referee. JSpecify/Checker-style @Nullable annotations plus IDE/SpotBugs analysis, and PIT's null-returning mutators, verify that the null row of your table is guarded by proof rather than by convention.

Floating point: boundaries that aren't even numbers

float and double trade the wrap-around seam for a different boundary zoo — special values and precision cliffs that ordinary equality logic mishandles:

Boundary valueThe trap
Double.NaNNaN != NaN is true — any comparison-based validation silently passes or fails sideways. Test with Double.isNaN, never ==.
±InfinityOverflow doesn't wrap here — Double.MAX_VALUE * 2 is Infinity, and it propagates through subsequent math without error.
-0.0-0.0 == 0.0 is true, but Double.compare(-0.0, 0.0) is negative and 1/-0.0 is -Infinity. Sorting and Map keys behave differently than ==.
Double.MIN_VALUENot the most negative double — it's the smallest positive one (≈4.9 × 10⁻³²⁴). The actual lower boundary is -Double.MAX_VALUE. A named-constant trap that BVA tables inherit if written from memory.
2⁵³ (9007199254740992)The integer-precision cliff: above it, d + 1.0 == d can be true. Any double used as a counter or ID crosses a boundary here.
0.1 + 0.2= 0.30000000000000004. Decimal fractions are themselves off-boundary in binary — compare with a delta (assertEquals(expected, actual, 1e-9)) or use BigDecimal.

A complete type-aware boundary table for a numeric input therefore has four layers: the spec boundaries (18 and 65), the storage boundaries (MIN_VALUE/MAX_VALUE and the seam), the reference boundary (null, whenever a boxed type appears anywhere in the data path), and for floating point the representation boundaries (NaN, infinities, −0.0, precision cliffs). The seven-point tables from the Foundations section apply unchanged at each layer — only the values differ.

§ Application · Server Side

BVA in Java: parameterized tests as boundary tables

JUnit 5's @ParameterizedTest is practically purpose-built for BVA: the boundary table you derive on paper becomes, line for line, a @CsvSource block. Each row is one boundary value plus its expected outcome; the framework runs the method once per row and reports each as an individual test case. The test suite is the boundary analysis, readable at a glance in a pull request. 

// The unit under test — V(G) = 3: two boundary predicates + 1
public class LoanEligibility {
    public boolean isEligibleAge(int age) {
        return age >= 18 && age <= 65;
    }
}

// Three-value BVA, one row per test point
class LoanEligibilityTest {

    private final LoanEligibility subject = new LoanEligibility();

    @ParameterizedTest(name = "age {0} → eligible: {1} ({2})")
    @CsvSource({
        // value, expected, role
        "17, false, min-1",
        "18, true,  min",
        "19, true,  min+1",
        "40, true,  nominal",
        "64, true,  max-1",
        "65, true,  max",
        "66, false, max+1"
    })
    void enforcesAgeBoundaries(int age, boolean expected, String role) {
        assertEquals(expected, subject.isEligibleAge(age));
    }

    // Type-level boundaries deserve their own rows
    @ParameterizedTest
    @ValueSource(ints = {Integer.MIN_VALUE, -1, 0, Integer.MAX_VALUE})
    void rejectsExtremeValues(int age) {
        assertFalse(subject.isEligibleAge(age));
    }
}

Making the complexity link operational

  • Measure first. SonarQube, PMD (CyclomaticComplexity rule), and Checkstyle all report V(G) per method; IntelliJ IDEA shows it inline via plugins. Sort descending — that's your BVA priority list.
  • Cover the paths, then the edges. JaCoCo branch coverage confirms both sides of every predicate ran; your @CsvSource rows confirm they flip at the right values. Different questions, both necessary.
  • Let mutation testing referee. PIT's CONDITIONALS_BOUNDARY mutator rewrites >= to > and similar. If a mutant survives, your suite is missing a boundary value — mutation testing is automated proof of whether your BVA was real or decorative.
  • Guard the enterprise seams. In layered Java systems, the same 18–65 rule often lives in Bean Validation (@Min(18) @Max(65)), service logic, and a database constraint. BVA each layer independently — boundary drift between layers is a classic integration defect.
Team habit worth adopting: require the boundary role column (min-1, min, nominal…) in every parameterized test's display name. Reviewers can then verify completeness of the boundary set without reconstructing the analysis — a missing max+1 row is visible in the diff.
§ Application · Devices

BVA in mobile: the device itself is a boundary

On mobile, BVA applies twice. The first application is familiar — input fields, form limits, payload sizes — and runs through the same seven-point tables as anywhere else, executed with Espresso (Android), XCUITest (iOS), or cross-platform tools like Appium and Maestro. The second application is what makes mobile distinct: the runtime environment is itself a bundle of bounded ranges, and each range has edges where apps break.

Environmental boundary classes

Input boundaries in native UI tests

// Android · Espresso + JUnit: username field spec'd at 3–20 chars
@RunWith(AndroidJUnit4::class)
class UsernameBoundaryTest {

    @Test fun rejectsTwoChars()      = assertUsername("ab",        valid = false) // min-1
    @Test fun acceptsThreeChars()    = assertUsername("abc",       valid = true)  // min
    @Test fun acceptsTwentyChars()   = assertUsername("a".repeat(20), valid = true)  // max
    @Test fun rejectsTwentyOne()     = assertUsername("a".repeat(21), valid = false) // max+1

    private fun assertUsername(input: String, valid: Boolean) {
        onView(withId(R.id.username)).perform(replaceText(input), closeSoftKeyboard())
        onView(withId(R.id.submit)).perform(click())
        onView(withId(R.id.username_error))
            .check(if (valid) matches(not(isDisplayed())) else matches(isDisplayed()))
    }
}
// iOS · XCUITest: same boundary table, Swift side
func testUsernameBoundaries() {
    let cases: [(input: String, valid: Bool)] = [
        (String(repeating: "a", count: 2),  false), // min-1
        (String(repeating: "a", count: 3),  true),  // min
        (String(repeating: "a", count: 20), true),  // max
        (String(repeating: "a", count: 21), false), // max+1
    ]
    for c in cases {
        let field = app.textFields["username"]
        field.tap(); field.clearText(); field.typeText(c.input)
        app.buttons["submit"].tap()
        XCTAssertEqual(app.staticTexts["username_error"].exists, !c.valid)
    }
}

Keep the pyramid in mind: the validation logic boundary tests belong at the unit level (fast, exhaustive tables), while UI-level frameworks verify the boundary experience — that the error actually renders, that a 21st character is blocked or flagged, that the layout survives the maximum-length string on the smallest screen. And because mobile flows lean heavily on conditional UI state (permission granted?, online?, first launch?), view models and controllers accumulate cyclomatic complexity fast — the same V(G) report that prioritizes your Java services will point at the mobile classes whose environmental boundary matrix most needs filling in.

§ Application · Front End

BVA in Angular: validators are boundaries as code

Angular's reactive forms make boundaries unusually explicit: Validators.min(18), Validators.max(65), Validators.minLength(3), Validators.maxLength(20) are boundary declarations sitting right in the component. That's a gift to the tester — the boundary table can be exercised against the FormControl directly, synchronously, with no DOM involved, because reactive form state updates are synchronous and fully under test control. 

// registration-form.component.ts
profileForm = this.fb.group({
  age:      [null, [Validators.required, Validators.min(18), Validators.max(65)]],
  username: ['',   [Validators.required, Validators.minLength(3), Validators.maxLength(20)]],
});

// registration-form.component.spec.ts — Jasmine/Karma or Jest, same shape
describe('age control boundaries (3-value BVA)', () => {
  let age: AbstractControl;

  beforeEach(() => { age = component.profileForm.get('age')!; });

  const table: [number, boolean, string][] = [
    [17, false, 'min-1'],
    [18, true,  'min'],
    [19, true,  'min+1'],
    [40, true,  'nominal'],
    [64, true,  'max-1'],
    [65, true,  'max'],
    [66, false, 'max+1'],
  ];

  table.forEach(([value, expectValid, role]) => {
    it(`${role}: value ${value} → valid=${expectValid}`, () => {
      age.setValue(value);
      expect(age.valid).toBe(expectValid);
    });
  });

  it('exposes the failing boundary in the error map', () => {
    age.setValue(17);
    expect(age.errors).toEqual({ min: { min: 18, actual: 17 } });
  });
});

Custom validators: where your boundaries actually live

Built-in validators cover simple ranges; real applications have compound boundaries — "end date after start date," "amount not exceeding remaining balance," "outside a blocked range." A custom validator is just a function from AbstractControl to an error map or null, which means it's a pure function you can BVA-test in isolation before it ever touches a form. Every if inside that validator is a decision point: complex custom validators are exactly where an Angular codebase's cyclomatic complexity concentrates (ESLint's complexity rule and SonarQube both measure TypeScript), and therefore exactly where the boundary tables should be richest.

Layered boundary checks in a front-end

  • Unit level (the bulk): validator functions and FormControl state, as above — fast enough to afford full three-value tables everywhere.
  • Component/DOM level: with Angular's TestBed and Harnesses, confirm the error message appears at min−1 and disappears at min — the user-facing half of the boundary. Remember HTML inputs return strings; the numeric coercion step is itself a boundary-rich seam ("18" vs 18 vs "" vs null).
  • E2E level (a thin skim): Cypress or Playwright, one representative boundary per critical form — proving client and server agree on where the edge is. A client that allows 65 while the API rejects it is a boundary-drift bug BVA at a single layer can't see.
  • Never trust the client: Angular validation is UX, not security. The authoritative boundary check must exist server-side (your Java layer) — and both sides must be tested against the same boundary table, kept in one place if you can manage it.
Full-stack symmetry: the same seven-row table drives the JUnit @CsvSource on the API and the forEach spec on the Angular form. Teams that generate both from one shared fixture file eliminate boundary drift by construction.
§ Force Multiplier

AI assistants as boundary engines — and complexity risks

Tools like GitHub Copilot, Claude, and Claude Code change the economics of both halves of this guide. Deriving a boundary table used to be careful manual work; an LLM can enumerate partitions, edge values, type seams, and null paths from a requirement or a method body in seconds. But the same tools also generate code — and the evidence says AI-written code trends toward higher cyclomatic complexity, which means more boundaries to test, written faster than anyone is testing them. Used deliberately, AI attacks both problems; used passively, it amplifies the second one.

Where AI genuinely improves boundary analysis

  • Boundary derivation from prose. Paste a requirement ("orders of $100 or more ship free; max 10 items; guests can't exceed $500") and ask for equivalence partitions plus a three-value BVA table. Frontier models do this reliably — comparative evaluations of unit-test generation found current-generation models (Claude 4-class, GPT-5-class, Gemini 2.5-class) identifying edge cases dependably, with the strongest producing two to three times as many boundary-relevant tests as older models — including the input-parameter boundaries weaker models skip.
  • Table-to-test translation. An agreed boundary table becomes a JUnit @CsvSource block, an Angular forEach spec, and an Espresso case list in one prompt each — the full-stack symmetry from earlier sections, generated instead of hand-copied. AI is excellent at exactly this kind of faithful cross-framework translation.
  • Type-layer completeness. The storage, reference, and representation boundaries from the type section — MIN/MAX seams, null-in-boxed-types, NaN and −0.0 — are precisely the systematic-but-tedious knowledge LLMs recall on demand. "List every type-level boundary value for this method's parameters" is a high-yield review prompt.
  • Reading complex conditionals. For a method dense with predicates, AI can walk the control flow, name each decision point, and state which input values flip it — effectively drafting the basis-path analysis that connects V(G) to concrete test inputs.

Why "table first, tests second" matters: asked simply to "write tests," LLMs bias toward plausible-looking happy paths. Forcing the boundary table into the output first makes the analysis reviewable before code exists — you approve the edges, then the assistant merely transcribes them. The complexity side: measure more, not less

The published evidence cuts both ways. On generation, Sonar's 2025 static-analysis study of 4,442 Java tasks across five LLMs found models producing functionally correct code flagged heavily for high cyclomatic and cognitive complexity — a predictable artifact of token-by-token generation optimizing local coherence over architecture. GitClear's multi-year analysis of real repositories found an eight-fold rise in duplicated code blocks as AI assistance spread, with copy-pasted lines overtaking refactored ones for the first time in 2024. In short: unreviewed AI code tends to raise V(G) and multiply the very boundary surfaces that need testing.

On remediation, the same class of tools is a capable fix: an empirical study feeding SonarQube-flagged functions to LLMs with structured prompts measured an average ~36% reduction in reported issues, with high-complexity functions successfully decomposed into helper methods. Guardrails: where AI-driven testing goes wrong

  • Hallucinated assertions. Benchmark evaluations consistently find generated tests asserting the wrong exception type or misreading error semantics — plausible tests that encode a false spec. Every generated boundary expectation needs the same review a junior engineer's PR would get; a wrong assertion is worse than a missing one, because it green-lights the bug.
  • Coverage theater. AI can flood a diff with tests that raise line coverage while never touching a seam. Judge generated suites by mutation score (does PIT's CONDITIONALS_BOUNDARY mutator die?) rather than coverage percent — mutation testing is the mechanical detector of decorative BVA.
  • Complexity creep in the code itself. Because AI output skews complex and generation is cheap, tighten the gates: Sonar's own guidance for AI-heavy codebases is stricter quality profiles — lower complexity thresholds, high new-code coverage, near-zero duplication. Every AI-added predicate is a boundary someone must now test; make the gate charge for it.
  • Ambiguity laundering. When a spec doesn't say whether 65 is inclusive, an LLM will pick one and write confident tests for it. Instruct assistants to surface ambiguous boundaries as questions (as in the prompt above) — the decision belongs to a human, and the ambiguity itself is the most valuable finding.
  • Skill atrophy is optional. Teams that treat the generated table as a review artifact — argued over in the PR like any design — keep boundary thinking sharp while shedding the transcription work. Teams that merge whatever appears lose the analysis skill the tool was supposed to amplify.
Bottom line: AI moves BVA from "technique you remember to apply" to "artifact you review by default" — and simultaneously raises the stakes on complexity measurement, because it writes branchy code faster than humans write boundary tests. The teams winning with these tools run the loop above; the metric, the table, and the mutant kill-rate are the three numbers that keep the assistant honest.
